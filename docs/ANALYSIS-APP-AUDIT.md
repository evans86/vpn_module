# Комплексный анализ приложения VPN (Laravel + Marzban)

**Цель:** выявить причины нестабильности, долгих загрузок конфигураций и оттока клиентов; предложить изменения без немедленного изменения кода.

**Референс API Marzban:** [Использование Marzban API](https://marzban-docs.sm1ky.com/tutorials/for_developer/) — Admin Token, User (Add/Get/Modify/Remove), Core Config, Subscription. Приложение использует те же HTTP-эндпоинты через Guzzle (PHP).

---

## 1. Резюме

| Проблема | Причина | Приоритет |
|----------|---------|-----------|
| Долгая загрузка конфигурации | Последовательные вызовы API к каждой панели в одном HTTP-запросе; refresh без кэша тянет все слоты с панелей | Критично |
| Долгое добавление ссылок в VPN-приложение | Подписка быстрая (из БД), но браузерная страница и «Обновить» вызывают getFreshUserLinks по каждому слоту (token + getUser, иногда getConfig + updateUser) | Критично |
| Нестабильность, «что-то ломается» | Нет retry для большинства вызовов Marzban; deleteUser при 500 не бросает исключение; updateUser без таймаута; старые слоты не удалялись при перевыпуске (уже исправлено) | Высокий |
| Воркер очереди отрубается | Хостинг убивает долгие процессы; нет обязательного автозапуска (cron/supervisor) | Высокий |
| «Костыли» | Заглушка при 500 в deleteUser; тушение ошибок; смешение sync/queue (addMissingProviderSlots то в запросе, то джобой) | Средний |

## 2. Архитектура и интеграция с Marzban

### 2.1 Текущая схема

- **Конфиг/подписка:** `VpnConfigController` → слоты из БД (KeyActivateUser + ServerUser). Для **подписки** (`?format=subscription` или VPN-клиент) ссылки берутся из `server_user.keys` (JSON) — **без вызовов к панелям**, быстро.
- **Страница в браузере:** сначала отдаётся shell с iframe/URL контента. Контент: кэш 180 сек или **buildConnectionData** → для каждого слота **getFreshUserLinks** (updateToken + getUser; для REALITY при малом числе links ещё getConfig + updateUser). Все вызовы **последовательно** в одном запросе.
- **Активация ключа:** синхронно в HTTP: getPanelsForActivation → для каждой панели addServerUser (token + createUser) → warmConfigSync (внутренний HTTP GET refresh до 60 сек). Очередь не используется.
- **Перевыпуск:** в джобе RenewKeyJob (но воркер может падать после первой же джобы). В renew: удаление пользователей на панелях, удаление KeyActivateUser в БД, создание новых слотов, warmConfigSync.

### 2.2 Вызовы Marzban API (по коду)

| Операция | Файл | Таймаут | Retry | Параллельно |
|----------|------|--------|-------|-------------|
| getToken | MarzbanAPI | 30/15 | нет | нет |
| createUser | MarzbanAPI | 20/10 | нет | нет |
| getUser | MarzbanAPI | 45/15 | нет | нет |
| updateUser | MarzbanAPI | **не задан** | нет | нет |
| deleteUser | MarzbanAPI | 45/15 | нет, при 500 — заглушка | нет |
| getConfig / modifyConfig | MarzbanAPI | 45/15 | только modify в MarzbanService (2 попытки) | нет |

Все вызовы к разным панелям/слотам выполняются **строго последовательно**. При 3 слотах и refresh: 3×(updateToken + getUser), при REALITY ещё 3×(getConfig + updateUser) — один медленный или «висящий» запрос блокирует весь ответ.

## 3. Корневые причины проблем

### 3.1 Производительность

1. **Последовательные вызовы к панелям**  
   При открытии страницы конфига или нажатии «Обновить» для каждого слота подряд: обновление токена панели + getUser (и при необходимости getConfig + updateUser). Время ответа = сумма задержек всех панелей. Одна «медленная» панель (таймаут 15–45 сек) блокирует всю страницу.

2. **warmConfigSync после активации и перевыпуска**  
   Синхронный внутренний HTTP GET на `/config/{id}/refresh` с таймаутом 60 сек. Увеличивает время ответа активации и время выполнения джобы перевыпуска; при недоступности конфиг-страницы джоба/запрос «висят».

3. **addMissingProviderSlots в HTTP-запросе**  
   При `syncMultiProvider = true` (refresh) или при `queue.default = sync` добавление недостающих слотов вызывается синхронно в том же запросе — дополнительные вызовы createUser к панелям в том же потоке.

4. **Отсутствие кэша для подписки по слотам**  
   Подписка отдаётся из БД (server_user.keys). Если ключи на панели обновлялись (например, после смены конфига Marzban), пользователь увидит старые ссылки до следующего refresh. При этом сам refresh тяжёлый (все слоты с панелей), поэтому «подогрев» не делается фоново.

### 3.2 Стабильность

1. **Нет единой политики retry для Marzban**  
   Retry есть только у modifyConfig (2 попытки в MarzbanService). Остальные методы при сетевой ошибке или 5xx сразу падают — пользователь видит ошибку или пустую конфигурацию.

2. **deleteUser при 500 не бросает исключение**  
   В коде возвращается «костыль» — ошибка удаления на панели может быть незаметна, при перевыпуске часть пользователей на панелях остаётся, в БД слоты удалены — рассинхрон.

3. **updateUser без таймаута**  
   При «зависании» панели запрос может висеть до дефолтного таймаута PHP/веб-сервера — блокируется весь запрос конфига.

4. **Воркер очереди**  
   На shared-хостинге процесс может быть убит (Exit 12 или по таймауту). Без cron/supervisor воркер не перезапускается — перевыпуски и другие джобы перестают обрабатываться.

5. **Зависимость от QUEUE_CONNECTION**  
   При `sync` тяжёлые операции (renew, addMissingSlots) выполняются в HTTP-запросе — риск таймаутов и нестабильности под нагрузкой.

### 3.3 Архитектурные и «костыльные» места

1. **Смешение синхронного и очередевого поведения**  
   addMissingProviderSlots вызывается то синхронно (при refresh или при sync), то через AddMissingSlotsForKeyJob. Разное поведение при разных настройках очереди усложняет отладку и предсказуемость.

2. **Дублирование логики выбора панелей**  
   getPanelsForActivation и addMissingProviderSlots оба опираются на config('panel.multi_provider_slots') и панели по провайдерам; лимит max_provider_slots добавлен в оба места — при изменении правил нужно помнить про оба пути.

3. **Жёсткие таймауты и лимиты в коде**  
   set_time_limit(180), memory 1024M, Http::timeout(60) для warmConfigSync, таймауты в MarzbanAPI разбросаны по методам — нет единого конфига «внешние API / панели».

## 4. Рекомендации (без изменения кода — план работ)

### 4.1 Критично: скорость отдачи конфигурации и ссылок

1. **Параллельные запросы к панелям**  
   При сборке конфига (buildConnectionData / getFreshUserLinks) запросы к разным панелям (token + getUser, при необходимости getConfig/updateUser) выполнять параллельно (например, через Guzzle Promise или отдельные HTTP-клиенты в циклах с ограничением concurrency), а не последовательно. Это сильно сократит время ответа страницы конфига и refresh.

2. **Кэш «свежих» ссылок по слотам**  
   После успешного getUser (и при необходимости updateUser) сохранять результат (links) в кэш с коротким TTL (например, 60–120 сек) по ключу `server_user_id` или `key_activate_id` + `panel_id`. При запросе конфига сначала отдавать из кэша; при промахе — запрос к панели и обновление кэша. Это уменьшит число обращений к панелям при частых открытиях страницы или подписки (если подписка начнёт использовать «свежие» ссылки из кэша).

3. **Убрать или ослабить warmConfigSync**  
   Варианты: не вызывать warmConfigSync после активации/перевыпуска; либо вызывать его асинхронно (отправить в очередь отдельную джобу «прогреть конфиг по key_id» без ожидания ответа). Это ускорит ответ активации и завершение джобы перевыпуска.

4. **Не вызывать addMissingProviderSlots синхронно в refresh**  
   При нажатии «Обновить» не добавлять недостающие слоты в том же запросе; ставить джобу AddMissingSlotsForKeyJob и отдавать конфиг по уже существующим слотам (или показывать «слоты обновляются» и дать ссылку на повтор через несколько секунд). Так refresh перестанет зависеть от нескольких createUser подряд в одном запросе.

### 4.2 Высокий приоритет: стабильность

5. **Единая обёртка вызовов Marzban с retry и таймаутами**  
   Все вызовы к Marzban (getToken, getUser, createUser, updateUser, deleteUser, getConfig, modifyConfig) выполнять через один слой (например, в MarzbanAPI или отдельном MarzbanClient) с: фиксированными timeout/connect_timeout из конфига; retry (например, 2 попытки с экспоненциальной задержкой) для сетевых ошибок и 5xx; логированием ошибок. Для updateUser явно задать таймаут.

6. **Корректная обработка deleteUser**  
   Убрать заглушку при 500: при ошибке удаления — логировать и пробрасывать исключение (или возвращать результат с флагом ошибки), чтобы перевыпуск и админка могли реагировать (повтор, ручное удаление на панели).

7. **Гарантированный запуск воркера очереди**  
   Настроить cron (или supervisor при наличии прав) для периодической проверки и запуска воркера (как в scripts/ensure-queue-worker.sh). Документировать обязательность очереди (database/redis) и воркера для перевыпуска и других фоновых задач.

8. **Мониторинг панелей и очереди**  
   Простая проверка: периодически (cron) убеждаться, что воркер запущен и что ключевые панели отвечают (getToken или лёгкий endpoint). При падении воркера — алерт или запись в лог/админку.

### 4.3 Средний приоритет: архитектура и поддержка

9. **Конфиг для внешних API и панелей**  
   Вынести в один конфиг (например, `config/panel.php` или `config/external_api.php`) таймауты, число повторных попыток и политику retry для Marzban. Использовать эти значения во всех вызовах (MarzbanAPI, MarzbanService).

10. **Единый путь «добавления слотов»**  
    Добавление недостающих слотов всегда через очередь (джобу), без синхронного вызова в refresh или при sync. Лимит max_provider_slots применять в одном месте (getPanelsForActivation и при добавлении слотов), чтобы не дублировать логику.

11. **Индексы и тяжёлые запросы**  
    Проверить схему key_activate (индексы по status, user_tg_id, finish_at для выборок и отчётов). При необходимости добавить индексы; проверить запросы с with/whereHas, чтобы не было N+1 при списках ключей и слотов.

12. **Документация по Marzban**  
    В проекте зафиксировать: какие эндпоинты Marzban используются (Admin Token, User Add/Get/Modify/Remove, Core Config), откуда они вызываются (MarzbanAPI, MarzbanService, VpnConfigController), какие таймауты и retry применяются. Ссылаться на [официальную документацию Marzban API](https://marzban-docs.sm1ky.com/tutorials/for_developer/).

## 5. Ожидаемый эффект

| Действие | Эффект |
|----------|--------|
| Параллельные запросы к панелям при сборке конфига | Сокращение времени загрузки страницы конфига и refresh в 2–3 и более раз при нескольких слотах |
| Кэш ссылок по слотам (короткий TTL) | Меньше запросов к панелям, быстрее повторные открытия конфига |
| Отказ от синхронного warmConfigSync / вынос в очередь | Быстрее ответ активации и завершение перевыпуска, меньше «зависаний» |
| Retry и таймауты для всех вызовов Marzban | Меньше «случайных» ошибок при кратковременных сбоях панели или сети |
| Корректная обработка deleteUser и автозапуск воркера | Предсказуемый перевыпуск и меньше рассинхрона между БД и панелями |

В совокупности это должно снизить время отклика конфигурации и добавления ссылок в VPN-приложения, повысить стабильность и уменьшить отток клиентов из-за долгих или неработающих сценариев.

## 6. Порядок внедрения (предложение)

1. **Фаза 1 (быстрые победы):** таймаут для updateUser; убрать заглушку в deleteUser; вынести warmConfigSync в джобу или отключить; настроить cron для воркера; документировать необходимость очереди.
2. **Фаза 2 (производительность):** параллельные запросы при сборке конфига; кэш ссылок по слотам; убрать синхронный addMissingProviderSlots из refresh.
3. **Фаза 3 (надёжность и порядок):** обёртка вызовов Marzban с retry и общим конфигом; единый путь добавления слотов через очередь; проверка индексов и N+1; внутренняя документация по Marzban.

После утверждения плана можно переходить к конкретным изменениям в коде по фазам.
